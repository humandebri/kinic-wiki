LLM Wiki
LLM を使って個人向け知識ベースを構築するためのパターン。 

これはアイデア文書です。あなた自身の LLM エージェント（たとえば OpenAI Codex、Claude Code、OpenCode / Pi など）にコピペして使うことを想定しています。目的は高レベルな発想を伝えることであり、具体的な実装はあなたのエージェントが、あなたと協調しながら組み立てていきます。

## 中核となる考え方

多くの人にとって、LLM とドキュメントの関係は RAG に近いものです。つまり、ファイル群をアップロードし、質問時に LLM が関連チャンクを取り出して回答を生成するやり方です。これは機能しますが、LLM は質問のたびに知識をゼロから再発見しています。5つの文書をまたいで統合しないと答えられないような微妙な質問をすると、毎回そのたびに関連断片を探し直して組み立て直さなければなりません。蓄積がありません。NotebookLM、ChatGPT のファイルアップロード機能、そして大半の RAG システムはこの方式です。

ここでの発想は違います。生の文書から質問時に取り出すだけではなく、LLM が永続的な wiki を段階的に構築し、保守していくのです。これは構造化され、相互リンクされた Markdown ファイル群であり、生のソースとあなたの間に位置します。新しいソースを追加したとき、LLM は単に後で検索できるようにインデックス化するのではありません。読み込み、重要情報を抽出し、既存の wiki に統合します。エンティティページを更新し、トピック要約を改訂し、新しいデータが古い主張と食い違う箇所を記録し、進化中の統合見解を補強または修正します。知識は毎回再導出されるのではなく、一度コンパイルされ、その後は最新状態に保たれます。

ここが決定的に違う点です。wiki は永続的で、蓄積していく成果物です。相互参照はすでに張られており、矛盾点はすでに指摘されており、統合見解はこれまで読んだものすべてをすでに反映しています。新しいソースを追加し、新しい質問をするたびに、wiki はより豊かになります。

あなた自身が wiki を書くことはありません。少なくとも、ほとんどありません。wiki を書き、保守するのは LLM です。あなたの役割は、ソースの収集、探索、そして適切な問いを立てることです。要約、相互参照、整理、帳尻合わせのような、知識ベースを時間とともに本当に使えるものにするための面倒な作業は、すべて LLM が担います。実際には、私は片側に LLM エージェント、もう片側に Obsidian を開いています。LLM は会話に基づいて編集を行い、私はその結果をリアルタイムで眺めます。リンクをたどり、グラフビューを確認し、更新されたページを読む。Obsidian は IDE、LLM はプログラマ、wiki はコードベースです。

これはさまざまな文脈に適用できます。たとえば次のようなものです。

個人利用：目標、健康、心理、自己改善の追跡。日記、記事、ポッドキャストのメモなどを整理し、時間をかけて自分自身の構造化された像を築いていく。
研究：数週間から数か月かけてあるテーマを深掘りする。論文、記事、レポートを読み、進化していく仮説とともに包括的な wiki を段階的に構築する。
読書：章ごとに整理し、登場人物、テーマ、プロットの筋、それらのつながりについてページを作る。読み終えるころには、豊かな補助 wiki ができあがる。たとえば Tolkien Gateway のようなファン wiki を考えてみてください。登場人物、場所、出来事、言語を扱う何千もの相互リンクされたページが、長年にわたりコミュニティのボランティアによって構築されています。読書しながら、LLM に相互参照と保守をやらせることで、個人的に似たようなものを作れます。
ビジネス／チーム：Slack スレッド、会議の文字起こし、プロジェクト文書、顧客との通話記録を流し込み、LLM が保守する社内 wiki。場合によっては人間が更新内容をレビューする運用もありえます。チームの誰もやりたがらない保守作業を LLM がやるので、wiki は最新のまま保たれます。
競合分析、デューデリジェンス、旅行計画、講義ノート、趣味の深掘りなど、時間とともに知識が蓄積していき、それを散在させず整理したい場面全般。

## アーキテクチャ

3つの層があります。

**Raw sources** — 厳選したソース文書のコレクション。記事、論文、画像、データファイル。これらは不変です。LLM は読むだけで、変更しません。これが真実の源泉です。

**The wiki** — LLM が生成した Markdown ファイル群のディレクトリ。要約、エンティティページ、概念ページ、比較、概要、総合的な見解など。LLM がこの層を全面的に管理します。ページを作成し、新しいソースが来たら更新し、相互参照を保守し、全体の整合性を保ちます。読むのはあなた、書くのは LLM です。

**The schema** — 文書（たとえば Claude Code なら CLAUDE.md、Codex なら AGENTS.md）で、wiki の構造、規約、ソース取り込み時・質問応答時・保守時のワークフローを LLM に指示します。これが中核的な設定ファイルです。これによって LLM は、単なる汎用チャットボットではなく、規律ある wiki 管理者になります。何がうまくいくかが見えてくるにつれ、あなたと LLM は時間をかけてこれを共進化させていきます。

## 運用

**Ingest（取り込み）**
新しいソースを raw collection に置き、LLM に処理させます。典型的な流れはこうです。LLM がソースを読み、重要ポイントについてあなたと議論し、wiki に要約ページを書き、index を更新し、関連するエンティティページや概念ページを wiki 全体で更新し、最後に log に記録を追記します。1つのソースが 10〜15 個の wiki ページに影響することもあります。私自身は、ソースを一度に1件ずつ取り込み、関与し続けるやり方を好みます。要約を読み、更新内容を確認し、何を重視すべきかを LLM に指示するわけです。ただし、監督を少なめにして複数ソースを一括取り込みすることもできます。自分のスタイルに合ったワークフローを作り、それを schema に文書化しておけば、以後のセッションでも使えます。

**Query（問い合わせ）**
wiki に対して質問します。LLM は関連ページを探し、読み込み、出典つきで回答を統合します。質問に応じて回答の形式はいろいろありえます。Markdown ページ、比較表、スライドデッキ（Marp）、チャート（matplotlib）、canvas などです。重要なのは、質の高い回答は新しいページとして wiki に戻せるという点です。比較、分析、発見したつながり。こうしたものは価値があり、チャット履歴の中に消えてよいものではありません。こうすることで、取り込んだソースと同じように、探索の成果も知識ベースの中で蓄積していきます。

**Lint（点検）**
定期的に、LLM に wiki の健全性チェックをさせます。見るべきものは、ページ間の矛盾、新しいソースによって上書きされた古い主張、被リンクのない孤立ページ、言及されているのに専用ページがない重要概念、欠けている相互参照、Web 検索で埋められそうなデータギャップなどです。LLM は、新たに調べるべき問いや探すべきソースの提案も得意です。これによって、wiki が大きくなっても健全さを保てます。

## インデックスとログ

wiki が大きくなると、LLM と人間の双方がナビゲートしやすくなるよう、2つの特別なファイルが役立ちます。役割は異なります。

**index.md** は内容志向です。wiki 内の全ページのカタログであり、各ページについてリンク、1行要約、必要なら日付やソース数などのメタデータも付けます。カテゴリ別（エンティティ、概念、ソースなど）に整理します。LLM は取り込みのたびにこれを更新します。質問に答えるときは、まず index を読んで関連ページを見つけ、その後に掘り下げていきます。これは中規模（約100ソース、数百ページ程度）では驚くほどよく機能し、埋め込みベースの RAG 基盤を用意せずに済みます。

**log.md** は時系列です。何がいつ起きたかを追記式で記録します。取り込み、問い合わせ、lint 実行などです。実用的なコツとして、各エントリを一定の接頭辞で始めるとよいです。たとえば `## [2026-04-02] ingest | Article Title` のようにしておけば、単純な unix ツールで解析可能になります。`grep "^## \[" log.md | tail -5` で直近5件を取れる、という具合です。log は wiki がどう進化してきたかのタイムラインを与え、LLM が最近何をやったかを把握する助けにもなります。

## 任意要素：CLI ツール

ある時点で、LLM が wiki をより効率的に扱えるようにする小さなツールを作りたくなるかもしれません。もっとも obvious なのは wiki ページ向けの検索エンジンです。小規模なら index ファイルだけで足りますが、wiki が成長すれば本格的な検索が欲しくなります。qmd は有力な選択肢です。Markdown ファイル向けのローカル検索エンジンで、BM25 とベクトル検索のハイブリッド検索に加え、LLM による再ランキングを、すべてローカル環境で実行できます。CLI もあり、LLM がシェルから呼び出せますし、MCP サーバとしても動くのでネイティブツールのように使わせることもできます。もっと単純なものを自作してもかまいません。必要が出た時点で、LLM に手伝わせて雑に検索スクリプトを書くこともできます。

## コツと小技

Obsidian Web Clipper は、Web 記事を Markdown に変換するブラウザ拡張です。ソースを raw collection に素早く入れるのに非常に便利です。

画像はローカルにダウンロードしておくとよいです。Obsidian の Settings → Files and links で “Attachment folder path” を固定ディレクトリ（例：`raw/assets/`）に設定します。そのうえで Settings → Hotkeys で “Download” を検索し、“Download attachments for current file” にホットキー（例：Ctrl+Shift+D）を割り当てます。記事を clip した後、そのホットキーを押せば画像がすべてローカルディスクに保存されます。必須ではありませんが、有用です。URL 切れに頼らず、LLM が画像を直接見て参照できるようになるからです。ただし、LLM はインライン画像つき Markdown を1回でそのまま読めるわけではありません。実際の回避策は、先に本文を読ませ、その後で参照画像の一部または全部を別途見せて追加コンテキストを与えることです。少し不格好ですが、十分実用的です。

Obsidian のグラフビューは、wiki の形を把握する最良の手段です。何が何につながっているか、どのページがハブか、どれが孤立しているかが見えます。

Marp は Markdown ベースのスライド形式です。Obsidian にはプラグインがあります。wiki の内容から直接プレゼン資料を生成するのに便利です。

Dataview は、ページのフロントマターに対してクエリを実行する Obsidian プラグインです。LLM が wiki ページに YAML frontmatter（タグ、日付、ソース数など）を追加するなら、Dataview で動的な表や一覧を生成できます。

wiki は単なる Markdown ファイルの git リポジトリです。バージョン履歴、ブランチ、共同作業の仕組みが最初から手に入ります。

## なぜこれが機能するのか

知識ベースを維持するうえで退屈なのは、読むことでも考えることでもありません。帳尻合わせです。相互参照の更新、要約の最新化、新しいデータが古い主張と矛盾する場合の記録、何十ものページの整合性維持。人間が wiki を放棄するのは、保守コストの増え方が価値の増え方を上回るからです。LLM は飽きないし、相互参照の更新を忘れないし、一度に15ファイル触ることもできます。保守コストがほぼゼロになるので、wiki は保守され続けます。

人間の役割は、ソースを選別し、分析を方向づけ、良い問いを立て、それが何を意味するかを考えることです。LLM の役割は、それ以外の全部です。

この発想は、ヴァネヴァー・ブッシュの Memex（1945年）と精神的に近いものです。個人的にキュレーションされた知識貯蔵庫であり、文書間に連想的な経路を持つものです。ブッシュの構想は、Web が実際にそうなった姿よりも、こちらに近いものでした。つまり、私的で、能動的に整理されており、文書そのものと同じくらい文書間のつながりが重要なものです。彼が解けなかったのは、誰がその保守をするのかという問題でした。そこを LLM が担います。

## 注記

この文書は意図的に抽象的です。説明しているのはアイデアであって、特定の実装ではありません。ディレクトリ構造、schema の規約、ページ形式、ツール群などは、あなたのドメイン、好み、使用する LLM に依存します。上で述べたものはすべて任意であり、モジュール式です。使えるものを採用し、不要なものは捨てればよい。たとえば、ソースがテキストだけなら画像処理は不要です。wiki が十分小さいなら index ファイルだけで足り、検索エンジンはいりません。スライドは不要で Markdown ページだけ欲しいかもしれません。まったく別の出力形式が欲しいかもしれません。正しい使い方は、この文書をあなたの LLM エージェントに渡し、一緒に自分に合う形へ具体化していくことです。この文書の仕事は、あくまでパターンを伝えることだけです。残りはあなたの LLM が組み立てればいい。
